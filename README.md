## Overview

This project implements an unsupervised machine learning system for detecting insider threats from user activity logs. It engineers six behavioural features from the CERT Insider Threat Dataset (v4.2), applies an Isolation Forest anomaly detection model, and produces interpretable risk classifications (Low / Medium / High) for security analyst review via a web-based dashboard.

---

## Core Features

- **Multi-source feature engineering** вЂ” six per-user daily behavioural features derived from logon, file, email, device, and HTTP logs
- **Unsupervised anomaly detection** вЂ” Isolation Forest trained on unlabelled data; no ground truth labels required
- **Rule-based risk scoring layer** вЂ” percentile thresholds + rolling z-score top-reason labelling for interpretable outputs
- **Web-based analyst dashboard** вЂ” ranked flagged-user table, anomaly score chart, and per-user behavioural detail view
- **Ground truth evaluation** вЂ” optional classification report (precision, recall, F1, false positive rate) against CERT answers
- **Modular pipeline** вЂ” five independent source modules that can be run and tested separately

### 4. Obtain the CERT Insider Threat Dataset

> вљ пёЏ **The dataset is not included in this repository** because the raw CSV files are too large for version control (several GB). You must download them separately and place them in `data/raw/` before running the pipeline.

This project uses the **CERT Insider Threat Dataset v4.2**, available from Carnegie Mellon University's Software Engineering Institute:

> https://doi.org/10.1184/R1/12841247.v1

Download the dataset (file: `r4.2.tar.bz2`) and extract the following CSV files into `data/raw/`:


The ground truth labels file (`insiders.csv`) is in a **separate** `answers.tar.bz2` archive, also available from the same download page. Extract it with:

```bash
tar -xjf answers.tar.bz2 -C data/raw --strip-components=1 answers/insiders.csv
```

Once all six files are in `data/raw/`, the pipeline is ready to run.

## Running the Pipeline

The pipeline is designed to be run step-by-step. Each step reads from and writes to `data/processed/`.

### Step 1 вЂ” Load and merge all event logs

```bash
python3 tests/test_events_table.py
```

Output: `data/processed/events_demo.csv`

### Step 2 вЂ” Engineer features

```bash
python3 tests/test_features.py
```

Output: `data/processed/features_user_day_demo.csv`

### Step 3 вЂ” Train Isolation Forest and score observations

```bash
python3 tests/test_model_iforest.py
```

Output: `data/processed/scored_user_day_demo.csv`

### Step 4 вЂ” Apply risk scoring layer

```bash
python3 tests/test_risk.py
```

Output: `data/processed/alerts_user_day_demo.csv`

### Step 5 вЂ” Evaluate against ground truth

```bash
python3 src/evaluate.py
```

Prints risk distribution, top anomalous users, and (if `insiders.csv` is present) a full classification report with precision, recall, F1-score, and false positive rate.

## Demo Mode

By default, the pipeline runs in **demo mode**, which filters to:
- Date range: **JanuaryвЂ“February 2010**
- User sample: **first 100 users alphabetically**

This allows fast iteration without processing the full 18-month dataset. To run on the full dataset, set `demo_mode=False` in `PipelineConfig`:

```python
cfg = PipelineConfig(
    data_dir=Path("data/raw"),
    demo_mode=False          # Process full dataset
)
```

## Known Limitations

- **Dataset not included in repo** вЂ” the CERT CSV files are too large for version control. Follow the Dataset Setup instructions above to download and place them in `data/raw/` before running.
- **Offline evaluation only** вЂ” the system processes static CSV files. Real-time stream processing is not implemented.
- **No authentication system** вЂ” the dashboard does not implement user login or access control.
- **Synthetic dataset** вЂ” results are evaluated on the CERT synthetic dataset; generalisation to real organisational environments has not been validated.
- **No model persistence** вЂ” the Isolation Forest is retrained on each pipeline run. Model saving/loading (e.g. via `joblib`) is not implemented.
- **HTTP log dependency** вЂ” if `http.csv` is absent from your CERT version, `website_visits_count` will default to 0 for all users.
- **Email log column variance** вЂ” some CERT versions use slightly different column names in `email.csv`. The `_safe_read_csv()` function tolerates this but may silently drop optional columns.

## Configuration Reference

Key parameters in `PipelineConfig` (`pipeline_load.py`):

| Parameter | Default | Description |
|---|---|---|
| `demo_mode` | `True` | Filter to date range and user sample |
| `date_start` | `2010-01-01` | Start date for demo window |
| `date_end` | `2010-02-28` | End date for demo window |
| `user_sample_size` | `100` | Number of users to include in demo mode |
| `after_hours_start` | `18` | Hour (24h) after which activity is flagged |
| `after_hours_end` | `8` | Hour (24h) before which activity is flagged |

Key parameters in `model_iforest.py`:

| Parameter | Default | Description |
|---|---|---|
| `contamination` | `0.05` | Expected proportion of anomalous observations |
| `n_estimators` | `200` | Number of isolation trees |
| `random_state` | `42` | Seed for reproducibility |
